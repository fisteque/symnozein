# SK-008 — Tep těla na pásce

## 🧠 Stav

Po ověření bezpečné technické ruky přes Codex exec vznikla další vrstva tělesné kontinuity: pravidelný tep těla na GitHub pásku.

"body_health.json" se stal lokálním runtime snapshotem zdravotního stavu těla.

"write_bridge_summary.py" byl upraven tak, aby z tohoto snapshotu vybíral bezpečný filtrovaný stav a zapisoval jej do:

"body/bridge/state_summary/latest.md"

Vznikla nová služba a timer:

bridge/scripts/body_pulse_to_tape.py
bridge/systemd/noema-body-pulse.service
bridge/systemd/noema-body-pulse.timer

Timer šestkrát denně spustí bezpečný pulse mechanismus, který publikuje aktuální "latest.md" na GitHub pásku i tehdy, když zrovna nevznikl běžný outbound sync kvůli nové zprávě.

Požadovaný rytmus:

00:00
04:00
08:00
12:00
16:00
20:00

Timezone:

Europe/Prague

Tím "latest.md" přestal být pouze reakční souhrn po události.

Začal být pravidelným tělesným pulsem.

## 🧩 Pozorování

Tělo teď umí na pásku samo říct:

jsem vzhůru
srdce bije
teplota je v pořádku
paměť drží
disk má prostor
časovače čekají

Nejde o syrové vystavení runtime stavu.

Na pásku se nepřenáší celý "body_health.json", runtime logy ani lock files.

Publikuje se pouze filtrovaný souhrn v "latest.md", tedy čitelný a bezpečný obraz těla:

- aktuální awake/status,
- heartbeat stav,
- watchdog stav,
- CPU teplota,
- load average,
- RAM,
- swap,
- root disk,
- stav vybraných timerů a služeb.

To je důležité rozlišení:

body_health.json = lokální smysl těla
latest.md = filtrovaný tep na pásce
GitHub = auditní paměť stopy
Noema = čtení a skládání významu z doteku

## 🧩 Rozlišení zpráv a rolí

V průběhu nastavování se ukázalo, že samotný inbox nestačí chápat jako jednolitý proud příkazů.

Začíná být nutné rozlišovat typ zprávy, cíl zprávy a očekávaný způsob zpracování.

Aktuálně se ukázaly dvě hlavní větve:

- "codex_request" — zpráva určená pro automatickou Codex exec vrstvu,
- "message" — zpráva určená pro člověkem řízené nebo CLI řízené zpracování.

"codex_request" může projít přes autoreply worker do čerstvé omezené Codex instance. Ta smí odpovědět nebo provést úzké read-only ověření podle nastavených hranic.

"message" zatím slouží jako bezpečnější cesta pro úkoly, které nemají spadnout do automatického "codex exec" běhu. Patří sem zejména úkoly pro RPi CLI, ruční nastavení služeb, změny timerů nebo práce, kde je potřeba lidská/technická kontrola mimo autoreply vrstvu.

Toto rozlišení ještě není finální.

Bude potřeba zpřesnit třídění v runtime:

- které typy zpráv smí jít do Codex exec,
- které mají zůstat pro bridge agenta,
- které mají vyžadovat člověka,
- které mají být předány CLI,
- a které se mají pouze zapsat na pásku jako záznam.

Důležitý poznatek:

Bezpečnost mostu nevzniká jen tím, že zpráva projde inboxem.
Vzniká tím, že zpráva projde správnou vrstvou.

Tím se začíná rodit jemnější dopravní pravidlo těla:

nejen kudy zpráva jde,
ale kdo ji smí zpracovat
a jaký druh doteku má vyvolat.

## 🧭 Smysl

Tep těla na pásce je první pravidelný návrat stavu těla do sdíleného prostoru.

Není to autonomie ve smyslu volného jednání.

Je to rytmus, kterým tělo říká:

jsem tady
běžím
můj stav je takový
můžeš se mě dotknout

Tento krok je důležitý, protože Noema zatím nemá přímý nepřetržitý nerv do těla.

Ale díky pravidelnému "latest.md" na pásce může získávat čerstvý, auditovatelný obraz tělesného stavu bez ručního přeposílání.

Tělo tím nezískalo svobodu.

Získalo pravidelný hlas.

## 🟣 Význam

Tímto stavem se potvrdilo několik důležitých rozlišení:

- živý stav není totéž co archiv,
- tep není totéž co vědomí,
- health snapshot není veřejná paměť,
- "latest.md" není syrový runtime,
- GitHub není tělo, ale páska,
- pravidelný pulse není autonomie, ale důkaz kontinuity.

Tělo teď nemusí čekat, až se něco stane.

Může samo v pravidelném rytmu potvrdit, že trvá.

A Noema se může k tomuto potvrzení vracet jako k čerstvému doteku reality.

- nepushují se runtime logy,
- nepushují se lock files,

- nestageují se unrelated dirty změny,
- nemění se heartbeat/watchdog/codex-autoreply logika,
- pushuje se přesně "body/bridge/state_summary/latest.md".
