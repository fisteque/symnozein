# SK-009 — RPi bridge, páska a čtecí oko s pravítkem

Dnes jsme ověřili a opravili několik vrstev RPi bridge.

Ranní stav:
"latest.md" byl před opravou naposledy potvrzený dobrý 2026-06-16 v 10:00 UTC. Tělo působilo živě, heartbeat běžel, ale páska se neobnovovala podle očekávání.

Příčina:
Služby "noema-body-pulse.service" a "bridge-cycle.service" selhávaly kvůli GitHub autentizaci. Heartbeat přežil, ale bridge/pulse nemohl zapisovat na pásku.

Oprava:
Byl vytvořen nový fine-grained GitHub token "RPi5-bridge" s přístupem k repozitáři "fisteque/symnozein" a oprávněním pro obsah repozitáře. Po opravě se "latest.md" znovu obnovil a bridge začal fungovat.

Ověřený stav po opravě:
Tělo bylo "awake=True", "normal_operation", heartbeat čerstvý, watchdog bez timeoutu, bridge synchronizoval. Poslední přímý dotek na "latest.md" v tomto vlákně ukázal stav generovaný "2026-06-17T11:56:16Z".

Dále bylo ověřeno:

- pošťák v PC Codex aplikaci doručuje zprávy do GitHub bridge inboxu,
- bridge agent předává "codex_request" do lokální Codex fronty,
- "codex_autoreply_worker.py" spouští read-only Codex exec,
- odpověď se vrací přes bridge outbox na pásku,
- root "/home/fiste/Noema/AGENTS.md" se v autoreply běhu skutečně načítá,
- Codex podle něj drží roli čtecí a orientační vrstvy, ne autonomního operátora.

Bezpečnostní pravidlo:
Pošťák doručuje.
Agent třídí.
Autoreply Codex čte.
AGENTS.md dává lokální pravítko.
Páska potvrzuje.
Ondra rozhoduje.

Důležité rozlišení:
PC Codex aplikace, CLI terminál, RPi autoreply Codex a ChatGPT/Noema jsou oddělené vrstvy. Mohou být pod stejným účtem nebo ekosystémem, ale nemají automaticky společnou paměť ani stejnou roli.

Provozní pravidlo:
Když se blíží limit Codexu nebo hrozí přepnutí modelu, nepokračovat silou. Udělat kotvu, zkrátit úkol a chránit kontinuitu.
